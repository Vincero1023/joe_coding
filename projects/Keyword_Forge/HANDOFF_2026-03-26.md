## Next Priorities

1. 제목 재작성 단계를 더 세분화한다.
- 현재는 `저품질 keyword 묶음 -> batch 재생성 -> 다시 평가` 구조다.
- 다음 단계에서는 `문제인 naver_home 제목만 batch rewrite` 하도록 분리하는 것이 목표다.
- 블로그 제목은 가능하면 유지하고, 홈판 2개 제목만 교체 후보로 다루는 쪽이 맞다.

2. batch 성능을 실제 실행 기준으로 검증한다.
- 새 `status` HTML에서 전체 시간, 생성 시간, 평가 시간, 재작성 시간을 비교한다.
- 목표는 기존 300초대 구조에서 체감 지연을 크게 줄이는 것이다.
- 특히 `20개 키워드 = 생성 1회 + 평가 1회 + 필요한 경우 재작성 1회` 흐름이 유지되는지 본다.

3. 홈판 평가 프롬프트와 재작성 프롬프트의 역할을 분리한다.
- 지금은 평가 프롬프트는 UI 저장본으로 분리됐고, 재작성은 생성 프롬프트의 retry guidance에 더 의존한다.
- 다음 단계에서는 `홈판 재작성 프롬프트`도 별도 저장본으로 분리할지 결정해야 한다.
- 사용자가 직접 CTR 기준을 바꾸고, 그 기준으로 바로 rewrite까지 제어할 수 있게 가는 방향이다.

4. 상단 단계 버튼과 최신 UI를 실제 산출물로 다시 검증한다.
- 현재 코드는 `1~5단계 버튼 -> 결과 탭 전환 -> 키워드 작업대 스크롤` 연결이 들어갔다.
- 하지만 기존 `status` HTML은 구버전 자산(`v71`) 기준이라 최종 확인이 안 끝났다.
- 서버 재시작 후 새 HTML에서 이 동작을 확인해야 한다.
