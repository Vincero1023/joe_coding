- 이 경로를 수정해 제목 자동 저장이 발생하면 최근 작업 이력도 함께 기록되도록 맞췄다.

- 제목 기본 타깃 모드는 백엔드 기준도 `단일 + 롱테일 V1`로 맞췄다. 다만 브라우저 `localStorage`에 저장된 예전 제목 설정이 있으면 그 값이 우선한다.

## 1. 현재 프로젝트 상태

- 로컬 실행 중심의 FastAPI 기반 키워드 파이프라인이다.
- 흐름은 `수집 -> 확장 -> 분석 -> 선별 -> 제목 생성 -> 전체 파이프라인`이다.
- 최근 작업은 `분석 실측 병렬화`, `노출도 지표 정리`, `제목 이슈 소스 분리`, `제목 export`, `운영 모드/세팅창`, `런타임 가드` 쪽에 집중됐다.
- 예약/큐 기본 기능은 이미 들어가 있다. 현재 남은 일은 메인 UI 연결, 운영 정책 고도화, 충돌/재시도 전략 정리 쪽이다.

## 3. 제목 생성 현재 상태와 한계

제목 품질이 아직 아쉽게 느껴질 수 있는 이유를 코드 기준으로 정리한다.

### 3-1. 지금 실제로 하는 것

- `app/title/ai_client.py`에서 키워드별로 네이버 검색 상위 제목을 가져온다.
- 여기서 다음 정도를 만든다.
  - `issue_terms`
  - `top_headlines`
  - `news_headlines`
  - `source_mix`
  - `news_count`
  - `community_reaction_summary`
  - `community_headlines`
- `issue_source_mode`가 `reaction` 또는 `mixed`일 때는 선택한 커뮤니티 도메인만 따로 집계해 반응 신호를 프롬프트에 붙인다.
- 그 결과를 프롬프트에 `live issue context`, `recent headlines`, `community reaction`, `community headlines` 형태로 붙인다.
- `app/title/main.py`는 이미 실행된 SERP 경쟁 요약이 있으면 그것을 재활용하고, issue source mode / community sources도 함께 정규화한다.

### 3-2. 지금 아직 안 하는 것

- 뉴스 본문 분석
- 댓글 반응 분석
- 커뮤니티 게시글 본문 분석
- 커뮤니티 댓글/추천수/반응량 분석
- 특정 사건의 사실관계 검증

즉 현재는 `실시간 이슈 반영`이 완전히 없는 것은 아니지만, 실제 수준은 `검색 상위 제목을 얕게 요약한 힌트`에 가깝다.

### 3-3. 왜 체감이 약할 수 있는가

- 검색 결과 제목만 보면 사람들이 왜 반응하는지의 맥락이 약하다.
- 뉴스/커뮤니티의 온도차를 분리하지 않는다.
- `지금 붙는 비교축`, `논쟁축`, `자주 같이 뜨는 고유명사`를 아직 충분히 구조화하지 않았다.
- 이 때문에 제목이 `지금 핫한 결`보다 `그럴듯한 SEO형 제목`에 머물 수 있다.

## 4. 제목 개선에 대한 마지막 조언 상세 기록

아래는 다음 작업 우선순위에 대한 권장안이다.

### 4-1. 결론

커뮤니티 반응을 참고하는 방향 자체는 맞다. 다만 바로 `커뮤니티 본문/댓글 크롤링`부터 들어가는 것은 유지비와 차단 리스크가 커서 비효율적일 가능성이 높다.

더 현실적인 순서는:

1. 뉴스 + 네이버 검색 상위 제목 분석을 먼저 강화
2. `뉴스형 / 반응형 / 공식발표형` 이슈를 분리
3. 그 다음 커뮤니티는 `검색 결과에 노출된 제목 기반`으로 제한적으로 도입
4. 마지막 단계에서만 특정 커뮤니티 본문/댓글 크롤링을 검토

### 4-2. 왜 바로 커뮤니티 크롤링이 비효율적인가

- 사이트별 파서가 다 다르다.
- 로그인, 성인, 차단, 봇 감지, 빈번한 레이아웃 변경 이슈가 많다.
- 루머성/감정성 콘텐츠가 많아 제목 품질이 흔들릴 수 있다.
- 예약모드까지 고려하면 장시간 유지보수 비용이 커진다.

### 4-3. 추천 1단계: 뉴스 + 네이버 반응형 SERP 강화

가장 먼저 할 일:

- 현재 SERP 요약을 더 깊게 읽도록 강화
- 제목을 `news / blog / cafe / official / commerce / external` 정도로 버킷화
- 버킷별로 분리된 신호를 프롬프트에 넣기

목표:

- `뉴스에서 도는 이슈`
- `카페/블로그에서 반복되는 반응`
- `공식 안내/변경`

이 3가지를 하나로 섞지 않고 나눠서 모델이 읽게 만드는 것

효과:

- 홈판 제목이 더 지금 시점의 결을 반영한다.
- 본문/댓글 크롤링 없이도 상당 부분 품질 개선이 가능하다.

### 4-4. 추천 2단계: 이슈 축 구조화

단순히 제목 몇 개를 넣는 것보다 아래를 구조화하는 것이 중요하다.

- 비교축
  - 예: 신형 vs 구형, 지역 A vs 지역 B, 정책 전후, 가격 전후
- 논쟁축
  - 예: 왜 붐비는지, 왜 기대와 다른지, 왜 체감이 엇갈리는지
- 반전축
  - 예: 겉으로는 뜨거운데 실제 지표는 약한지, 반대로 관심은 낮아도 전환은 높은지
- 동반 키워드
  - 예: 인물명, 브랜드명, 정책명, 서비스명, 이벤트명

지금은 이 구조화가 약해서 제목이 넓게 퍼지고, 찔러 들어가는 느낌이 부족할 수 있다.

### 4-5. 추천 3단계: 커뮤니티는 제목 기반부터

바로 본문/댓글을 긁는 대신, 우선은 검색 결과에 노출된 커뮤니티 제목만 따로 추려도 된다.

예시 방향:

- `cafe.naver.com`
- `blog.naver.com`
- `post.naver.com`

메뉴가 필요하면:

```powershell
run_local.bat menu
```

### 5-2. Creator Advisor / 네이버 로그인 관련 주의

- Creator Advisor 세션은 로컬 브라우저 의존성이 강하다.
- 다른 컴퓨터에서는 기존 세션 파일을 그대로 믿지 말고 `전용 로그인 브라우저 열기`로 다시 로그인하는 것이 안전하다.
- `.local/naver_playwright/...` 아래 내용은 로컬 환경에 종속될 수 있다.
- 루트 `*.credentials.json`은 레거시 fallback만 남기고, 새 시크릿 파일은 `.local/credentials/` 아래에 두는 쪽으로 정리했다.

### 5-3. 프롬프트 저장본 / 운영 설정 주의

중요:

- 프롬프트 저장본은 현재 브라우저 `localStorage` 기반이다.
- 운영 설정 드로어 값도 기본적으로 브라우저 로컬 저장본을 사용하고, 저장 후 서버 런타임에 반영된다.
- 제목 export 결과 파일은 기본적으로 프로젝트 루트의 `output/` 아래에 생성된다.
- 즉 다른 컴퓨터에서 브라우저를 새로 열면:
  - 프롬프트 저장본 목록
  - 현재 활성 프롬프트 저장본
  - 브라우저 로컬에만 있던 세팅
  는 자동으로 따라오지 않는다.

반드시 기억할 점:

- 코드 변경은 프로젝트 파일에 남아 있으므로 그대로 따라온다.
- 브라우저 저장 상태는 자동 동기화되지 않는다.

필요하면 다음 작업으로 `프롬프트 저장본 export/import` 기능을 추가하는 것이 좋다.

### 5-4. 운영 설정 API 확인

런타임 운영 상태 확인/저장 엔드포인트:

- `GET /settings/runtime`
- `POST /settings/runtime`
- `POST /settings/runtime/reset-guards`

예약모드를 만들 때는 이 설정을 읽어 스케줄 실행 전에 작업 상한/보호 상태를 확인하면 된다.
